## 1. Architectural Blueprint: Layered Microservices

The system is decomposed into specialized microservices, each optimized for its specific function, adhering to the **Command Query Responsibility Segregation (CQRS)** pattern.

### Core Services:

1.  **Client Gateway Service (Edge Layer):** Handles initial connection, authentication (JWT validation), rate limiting, and routing requests to the appropriate backend service.
2.  **Real-Time Collaboration Service (The Hot Path):** Manages persistent WebSocket connections, handles incoming user edits, applies CRDT logic, and broadcasts changes to other connected users. This service is optimized for low latency.
3.  **Document Persistence Service (The Write Path):** Handles all durable writes to the primary database, manages document versioning via Event Sourcing, and ensures strong consistency for metadata and permissions.
4.  **Presence Service (The State Path):** Manages user presence (typing indicators, online/offline status) using Redis for fast, eventually consistent state.
5.  **Notification/Event Service (The Asynchronous Path):** Acts as the central broker interface, consuming events from the message queue and triggering downstream actions (e.g., sending notifications, updating search indexes).

### Data Flow Diagram (Critical Path)

The flow is split into two paths: the **Real-Time Edit Path** (low latency) and the **Persistence/Event Path** (durable).

**A. Real-Time Edit Path (P95 < 100ms)**

1.  **Client $\rightarrow$ CDN/WAF $\rightarrow$ Client Gateway:** Client establishes a persistent **WebSocket** connection to the Real-Time Collaboration Service.
2.  **Client $\rightarrow$ Real-Time Service:** Client sends an **Edit Operation** (CRDT operation) over the WebSocket.
3.  **Real-Time Service:**
    *   **Validation & CRDT Application:** The service validates the operation and applies the CRDT logic locally to the document state.
    *   **Presence Update:** Updates the user's presence status in **Redis** (eventually consistent).
    *   **Event Publishing:** Publishes the *committed* CRDT operation as an immutable event to the **Message Queue (Kafka)**.
    *   **Broadcast:** Broadcasts the resulting state change (or the operation itself) to all other connected clients subscribed to that document.
4.  **Other Clients $\leftarrow$ Real-Time Service:** Receive the broadcast and apply the operation to their local CRDT state, achieving near real-time synchronization.

**B. Persistence/Event Path (Durable & Consistent)**

1.  **Message Queue (Kafka):** The committed CRDT operation event is durably stored in a Kafka topic.
2.  **Document Persistence Service (Consumer):** Consumes the event from Kafka.
3.  **Database Write:** The service applies the event to the **PostgreSQL** database, updating the document's version history (Event Sourcing) and ensuring strong consistency for metadata/permissions.
4.  **Read Model Update:** The service updates the optimized **Read Model** (e.g., a denormalized view in PostgreSQL or a search index) for fast query access.

## 2. Key Design Decisions & Implementation Details

### A. Conflict Resolution Strategy: CRDTs

**Decision:** Implement **Conflict-free Replicated Data Types (CRDTs)** for the document content itself.

**Rationale:** CRDTs are superior to Operational Transformation (OT) for this scale because they inherently support **decentralized synchronization** and **offline editing**. They resolve conflicts mathematically by ensuring that the merge operation is commutative, meaning the final state is guaranteed regardless of the order in which operations arrive across distributed replicas. This directly addresses the requirement for offline edits and high availability.

**Implementation:**
*   Each character or text segment is assigned a unique, immutable identifier (e.g., a combination of User ID and a Lamport timestamp or a unique site ID).
*   Operations are defined by these IDs and positional metadata, allowing replicas to merge operations without needing a central sequencer.

### B. Storage Choices

| Data Type | Storage Choice | Rationale | Consistency Model |
| :--- | :--- | :--- | :--- |
| **Document Content (Working State)** | In-Memory/CRDT State (Client/Real-Time Service) | Lowest latency for local editing. | Eventual (Converges to Strong) |
| **Document Metadata (Permissions, Version History)** | **PostgreSQL** (Primary Relational Store) | Provides ACID properties, strong consistency, and transactional integrity for critical data. | Strong Consistency |
| **Real-Time Presence (Cursor, Typing)** | **Redis** (Distributed Cache/Store) | Extremely fast read/write for ephemeral data. Used for session tracking and presence state. | Eventual Consistency |
| **Large Media/Attachments** | **Object Storage (e.g., AWS S3)** | Highly durable, scalable, and cost-effective for unstructured binary data. | Eventual Consistency |
| **Event Log/History** | **Kafka** (Event Streaming Platform) | Durable, ordered, and replayable log for Event Sourcing. | Durable/Ordered |

### C. Caching Strategy

**Goal:** Minimize latency for document retrieval and presence checks.

*   **Read-Through/Cache-Aside:** The **Client Gateway** and **Document Persistence Service** will use a distributed cache (Redis) for frequently accessed document metadata (e.g., document structure, permissions).
*   **Cache Invalidation:** Invalidation is driven by the **Event Stream**. When the Document Persistence Service successfully commits a change to PostgreSQL, it publishes a `DocumentUpdated` event to Kafka. The Cache Service consumes this event and explicitly invalidates or updates the corresponding cache entry. This ensures cache coherence is driven by the source of truth (the event log).
*   **Cache Stampede Prevention:** Implement **probabilistic early expiration** and use **request coalescing** at the Gateway level to prevent multiple requests for the same document from hitting the database simultaneously.

### D. Message Queue & Event Design

**Tool:** **Apache Kafka** is the chosen broker due to its high throughput, durability, ordered delivery guarantees, and ability to support replayability, which is crucial for Event Sourcing.

**Design:**
*   **Topics:** Dedicated topics for different event types (e.g., `document_edits`, `user_presence`, `system_alerts`).
*   **Edit Propagation:** The core flow uses the `document_edits` topic. This ensures that every committed edit is recorded sequentially and can be replayed by any service (e.g., for analytics or rebuilding a read model).
*   **Dead-Letter Queues (DLQs):** Any consumer service (e.g., the Persistence Service) that fails to process an event after a defined number of retries will route the message to a dedicated DLQ topic. This prevents message loss and allows for manual inspection and recovery of corrupted events.

### E. Rate Limiting and Backpressure

*   **Rate Limiting (Edge):** Implemented at the **Client Gateway Service** using a **Token Bucket** algorithm, keyed by User ID and IP address, to protect the system from basic flooding and abuse.
*   **Backpressure (Internal):** Implemented between the **Real-Time Service** and the **Message Queue**. If the Kafka consumer lag exceeds a threshold, the Real-Time Service will temporarily throttle the rate at which it publishes events, preventing the system from being overwhelmed by its own output.

### F. Failure Handling and Resilience

This is a critical cross-cutting concern, relying heavily on the **Circuit Breaker** and **Retry** patterns.

1.  **Retries with Exponential Backoff & Jitter:** Applied to all external service calls (e.g., calling the Presence Service or external APIs). This prevents retry storms by introducing random delays.
2.  **Circuit Breakers:** Implemented within the **Client Gateway** and between microservices. If the Document Persistence Service shows a high failure rate (e.g., 500 errors), the circuit opens, immediately failing requests to that service and routing them to a fallback mechanism (e.g., serving stale data from the cache or returning a graceful error).
3.  **Idempotency:** All consumers reading from Kafka must be **idempotent**. They must use the unique Event ID to check if the operation has already been processed, ensuring that retries do not result in duplicate database writes.
4.  **Disaster Recovery (DR):**
    *   **Regional Failover:** Implement an **Active/Active** or **Active/Passive** setup across multiple geographic regions. The PostgreSQL cluster must use asynchronous or synchronous replication depending on the RPO/RTO target for the metadata.
    *   **RPO/RTO:** Define clear Recovery Point Objective (RPO) and Recovery Time Objective (RTO). For critical metadata, aim for near-zero RPO via synchronous replication across regions.
    *   **Data Backup:** Regular, immutable backups of the PostgreSQL cluster and Kafka logs must be maintained in separate, isolated storage.

### G. Scaling Strategy

*   **Horizontal Scaling:** All stateless services (Client Gateway, Real-Time Service) are designed to be **stateless** and scaled horizontally via Kubernetes/Autoscaling based on CPU utilization and WebSocket connection counts.
*   **Database Scaling (Sharding):** Implement **Horizontal Sharding** on the **PostgreSQL** database, likely using the **Document ID** or a derived Tenant ID as the shard key. This distributes the write load and allows the system to scale writes linearly.
*   **Read Scaling:** Achieved via **Read Replicas** of PostgreSQL and the aggressive use of **Redis** for presence and cached read models.

### H. Rollout Plan (Phased Approach)

1.  **Phase 1: Core MVP (Focus on Consistency & Basic Sync):**
    *   Implement the core CRDT logic and the basic WebSocket connection.
    *   Establish the PostgreSQL/Event Sourcing pipeline for *single-user* document persistence.
    *   Implement basic rate limiting and basic circuit breakers.
2.  **Phase 2: Scale & Resilience (Focus on Distribution):**
    *   Implement Kafka for event propagation.
    *   Introduce Redis for Presence and Caching.
    *   Implement horizontal sharding for the PostgreSQL layer.
    *   Implement full DR procedures (failover drills).
3.  **Phase 3: Global Scale & Optimization (Focus on Performance):**
    *   Deploy multi-region active/active infrastructure.
    *   Optimize CRDT performance based on real-world latency metrics.
    *   Refine caching invalidation strategies and implement advanced load shedding.
