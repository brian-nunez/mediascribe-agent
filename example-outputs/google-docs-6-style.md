This design addresses the extreme requirements of a global, real-time collaborative editor, focusing on achieving low latency (P95 < 100ms local, < 300ms remote) while guaranteeing data integrity and resilience against regional failures.

The core architectural decision revolves around the conflict resolution strategy, which dictates the trade-off between strong consistency (for critical metadata) and eventual consistency (for real-time presence and text synchronization).

## Architectural Overview: Hybrid CRDT/Event Sourcing Model

The proposed architecture is a **Hybrid Model** combining **Conflict-free Replicated Data Types (CRDTs)** for the document content synchronization and **Event Sourcing (via Kafka)** for guaranteed persistence and auditability. This allows the system to achieve high availability and offline editing capabilities (Eventual Consistency) while using a strongly consistent database for critical metadata (Strong Consistency).

### Core Components:

1.  **Client Layer:** Manages local state, generates CRDT operations, handles offline storage, and maintains WebSocket connections.
2.  **Edge Layer:** CDN, WAF, and API Gateway for security, routing, and TLS termination.
3.  **Real-Time Synchronization Layer:** WebSocket servers responsible for broadcasting CRDT operations and managing user presence.
4.  **Core Service Layer:** Domain services handling authentication, permissions, and document metadata (where Strong Consistency is enforced).
5.  **Persistence & Eventing Layer:** Kafka acts as the immutable event log (Event Sourcing), ensuring no committed edit is ever lost.
6.  **Data Stores:** A hybrid approach using a distributed NoSQL store for document state and a relational DB for transactional metadata.

## End-to-End Request/Data Flow

### 1. Real-Time Edit Flow (Low Latency Path)

This path is optimized for the P95 < 100ms local acknowledgement.

1.  **Client Action:** User types a character $\rightarrow$ Client generates a **CRDT Operation** ($Op$).
2.  **Local Application:** Client applies $Op$ immediately to its local document state (instant visual feedback).
3.  **Transmission:** Client sends $Op$ via a **WebSocket** connection to the nearest **Real-Time Sync Server**.
4.  **Server Validation & Broadcast:** The Sync Server validates the $Op$ and immediately broadcasts it to all other connected clients for that document.
5.  **Remote Application:** Receiving clients apply the $Op$ using the CRDT merging logic, updating their local state.
6.  **Presence Update (Eventual):** The Sync Server publishes a lightweight **Presence Event** (e.g., `UserX_Typing`) to Kafka for eventual consistency across the system.

### 2. Persistence & Durability Flow (Guaranteed Commit Path)

This path ensures that committed edits are never lost.

1.  **Operation Ingestion:** The Real-Time Sync Server receives the validated $Op$.
2.  **Event Sourcing:** The server writes the $Op$ as an immutable event to a dedicated **Kafka Topic** (e.g., `document_edits`). This is the **Source of Truth**.
3.  **Database Update (Asynchronous):** A dedicated **Persistence Consumer** service reads the event from Kafka and applies the change to the primary document store (NoSQL/Sharded DB).
4.  **Metadata Update (Strong Consistency):** If the edit involves a permission change or user profile update, a separate service writes the change to the **Strongly Consistent Database** (e.g., PostgreSQL).

### 3. Offline Edit Synchronization Flow

1.  **Offline Work:** User makes edits while disconnected. The client stores all generated $Op$s locally in IndexedDB.
2.  **Reconnection:** Client establishes a WebSocket connection.
3.  **Synchronization:** The client sends its backlog of local $Op$s to the Sync Server.
4.  **Server Integration:** The Sync Server validates the backlog and publishes these operations to the Kafka stream, allowing the system to merge them deterministically with concurrent remote operations.

## Detailed Design Decisions

### 1. Conflict Resolution Strategy: CRDTs (Primary) + Strong Consistency (Metadata)

*   **Document Content:** **CRDTs** are chosen for the document text itself. This provides the necessary mathematical guarantee that concurrent edits will converge to the same state, supports offline editing, and minimizes server-side transformation logic, which is complex and error-prone.
*   **Metadata (Permissions, Sharing):** **Strong Consistency** is enforced here. Any operation that modifies user roles, permissions, or sharing links must be routed through a service that uses a strongly consistent database (e.g., PostgreSQL with synchronous replication) to prevent security or access control failures.

### 2. Storage Choices

| Data Type | Storage Technology | Rationale | Scaling Strategy |
| :--- | :--- | :--- | :--- |
| **Document Content (CRDT State)** | Distributed NoSQL (e.g., Cassandra or a specialized Document Store) | Optimized for high-volume writes and horizontal scaling. CRDT state is stored as the current merged document structure. | **Horizontal Sharding** based on `Document ID`. |
| **Event Log (Source of Truth)** | Apache Kafka | Immutable, durable, ordered log for all edits and system events. Guarantees "never lose committed edits." | Partitioned by `Document ID`. |
| **User/Permission Metadata** | Relational Database (e.g., PostgreSQL) | Required for ACID properties and strong consistency for critical security/access control data. | **Vertical Scaling** initially, potentially **Sharding** by Tenant ID later. |
| **Presence/Cursor State** | Distributed Cache (Redis) | Extremely low-latency reads/writes for ephemeral data like cursor positions and online status. | Key-value store, sharded by `User ID` or `Document ID`. |

### 3. Real-Time Sync Design (WebSockets & Presence)

*   **Protocol:** **WebSockets** are mandatory for the low-latency, persistent, bidirectional communication required for real-time text updates and presence.
*   **Presence Tracking:** Use a **Hot Beat** mechanism. Clients send periodic pings to the Sync Server. The server tracks the last ping time. If a timeout occurs, the user is marked offline. This provides a reliable, low-overhead way to manage the eventually consistent presence state.
*   **Latency Management:** The P95 remote propagation target of 300ms is met by:
    *   Minimizing payload size (sending only CRDT operations, not full documents).
    *   Using geographically distributed **WebSocket servers** (regional deployment).
    *   Leveraging **CDN** for serving static assets and initial document loads.

### 4. Message Queue/Event Design (Kafka Backbone)

**Kafka** is the central nervous system, providing decoupling and durability.

*   **Topics:**
    *   `document_edits`: High-throughput topic for all CRDT operations (Source of Truth).
    *   `user_presence`: Topic for eventual consistency updates (cursor, typing indicators).
    *   `system_events`: For non-document related events (e.g., user login, permission changes).
*   **Reliability:** Use **At-Least-Once Delivery** as the default, with consumers implementing **Idempotency** checks (using unique event IDs) to ensure that processing a message twice does not corrupt the document state.
*   **Backpressure:** Kafka's inherent buffering acts as a natural backpressure mechanism. If consumers slow down, the queue depth increases, signaling the need for the consumer services to scale horizontally.

### 5. Cross-Cutting Concerns

#### Rate Limiting & Backpressure
*   **Rate Limiting:** Implement **Token Bucket** algorithms at the **API Gateway** level for all external requests (login, document creation) and per-user limits on the WebSocket connection layer to prevent abuse and flooding.
*   **Backpressure:** Managed primarily by the **Kafka Broker** and **Consumer Scaling**. If the Persistence Consumer cannot keep up, the queue depth grows, which signals the need to auto-scale the consumer service instances.

#### Reliability and Failure Control
*   **Circuit Breakers:** Implement **Circuit Breakers** (e.g., using a Service Mesh or library like Resilience4j) on all service-to-service calls (especially between the Sync Server and the Strong Consistency DB) to prevent cascading failures if a dependency becomes slow or unresponsive.
*   **Retries:** Implement **Retries with Exponential Backoff and Jitter** for transient network errors, but strictly limit the number of retries to prevent retry storms.
*   **Dead-Letter Queues (DLQs):** Any message that fails processing after the maximum retry attempts is routed to a DLQ for manual inspection, ensuring no committed edit is lost due to a transient consumer bug.

#### Observability
*   **Distributed Tracing:** Implement **OpenTelemetry** across all services. Every request (HTTP, WebSocket message, Kafka event) must carry a **Correlation ID** to trace the entire flow from client input through the CRDT application, Kafka, and persistence layers.
*   **Metrics:** Implement **RED Metrics** (Rate, Errors, Duration) for every service, focusing heavily on Kafka consumer lag and WebSocket connection health.
*   **Alerting:** Use **Composite Alerts** based on **SLOs** (e.g., alert if Kafka consumer lag exceeds 5 seconds for more than 60 seconds) rather than simple threshold alerts.

#### Disaster Recovery (DR)
*   **Regional Strategy:** Implement an **Active/Active** or **Active/Passive** multi-region deployment.
*   **Data Replication:** The Kafka cluster must be deployed across multiple regions with cross-region replication enabled. The NoSQL document store must also have cross-region replication configured.
*   **RPO/RTO:** Define strict Recovery Point Objectives (RPO) and Recovery Time Objectives (RTO). For critical data, aim for near-zero RPO via synchronous replication for the Strong Consistency DB, and rely on Kafka's durability for the CRDT stream.
*   **Testing:** Conduct mandatory, regular **Chaos Engineering** drills to simulate regional failures and verify that the failover mechanisms (load balancer rerouting, Kafka leader election) function as expected.

#### Rollout Plan
1.  **Phase 1: Core Infrastructure (MVP):** Deploy the Kafka backbone, the basic NoSQL document store, and the WebSocket Sync Server in a single region. Focus on proving the CRDT synchronization and basic persistence.
2.  **Phase 2: Consistency Layer:** Introduce the Strongly Consistent Database for metadata and implement the Event Sourcing consumer to ensure data integrity.
3.  **Phase 3: Global Scale & Resilience:** Deploy the system across multiple geographic regions. Implement multi-region Kafka replication and configure global load balancing.
4.  **Phase 4: Optimization & Hardening:** Implement advanced features (complex search indexing, advanced analytics pipelines) and conduct comprehensive Chaos Engineering to validate the resilience patterns (Circuit Breakers, DR).
