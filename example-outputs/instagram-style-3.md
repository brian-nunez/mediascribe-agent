This design outlines a highly distributed, event-driven architecture tailored to handle the extreme scale, high read/write throughput, and conflicting consistency requirements of a global social media platform.

## 1. Requirements, Assumptions, and Core Trade-offs

### 1.1 Requirements Summary
| Requirement | Target Metric | Architectural Implication |
| :--- | :--- | :--- |
| **Scale** | 500M MAU / 50M DAU | Mandatory Horizontal Scaling, Sharding, and Multi-Region Deployment. |
| **Media Uploads** | 20K/sec peak | Asynchronous processing, Direct-to-Object Storage, high-throughput ingestion pipeline. |
| **Feed Reads** | 2M/sec peak | Aggressive Caching (CDN, Distributed Cache), Fanout-on-Read/Write strategy. |
| **Feed Latency** | P95 < 300ms | Low-latency data access, optimized read paths. |
| **Account Data** | Strong Consistency | Relational/Strongly Consistent Database for user profiles, authentication, and security. |
| **Likes/Comments** | Eventual Consistency | NoSQL/Eventually Consistent Store for high-volume, non-critical interactions. |
| **Availability** | Survive regional failure | Multi-Region Active/Active deployment, Circuit Breakers, Load Shedding. |

### 1.2 Core Architectural Trade-offs
The design explicitly manages the CAP theorem trade-offs across different subsystems:

*   **User Data (Profiles, Auth):** Prioritize **Consistency (C)**. We will use a strongly consistent database topology.
*   **Social Interactions (Likes, Comments):** Prioritize **Availability (A)** and **Latency (L)**. We will use eventual consistency models.
*   **Media Delivery:** Prioritize **Availability (A)** and **Latency (L)**. We will rely heavily on CDNs and Object Storage.

## 2. End-to-End Request/Data Flows

### Flow A: Media Upload (High Write Throughput)
This flow is designed to be asynchronous and decoupled to meet the P95 < 1s acknowledgement requirement.

1.  **Client $\rightarrow$ API Gateway:** Client requests an upload session.
2.  **API Gateway $\rightarrow$ Auth/Rate Limiter:** Authentication and initial rate limiting checks.
3.  **API Gateway $\rightarrow$ Upload Service:** Service generates a **Pre-Signed URL** for the Object Storage bucket.
4.  **API Gateway $\rightarrow$ Client:** Returns the Pre-Signed URL.
5.  **Client $\rightarrow$ Object Storage (Direct):** Client uploads the large media file directly to the Object Storage (e.g., AWS S3). This bypasses the application servers, handling the 20K/sec load efficiently.
6.  **Object Storage $\rightarrow$ Event Stream (Kafka):** Upon successful upload, the Object Storage triggers an event (e.g., `MediaUploaded`) published to a Kafka topic.
7.  **Kafka $\rightarrow$ Processing Workers:** Dedicated media processing workers consume the event to perform transcoding, thumbnail generation, and metadata indexing.
8.  **Workers $\rightarrow$ Database/Cache:** Workers update the metadata in the strongly consistent database and populate the read-optimized cache layer.

### Flow B: Home Feed Read (High Read Throughput)
This flow is optimized for low latency and high availability.

1.  **Client $\rightarrow$ CDN:** Client requests the home feed URL. The CDN serves cached static assets and potentially cached feed fragments.
2.  **CDN $\rightarrow$ API Gateway/Load Balancer:** Request hits the ingress layer.
3.  **API Gateway $\rightarrow$ Feed Service:** The service checks the **Distributed Cache** (e.g., Redis) for the user's pre-computed feed.
4.  **Cache Hit:** If the feed is present, it is returned immediately ($\ll 300\text{ms}$).
5.  **Cache Miss (Fanout-on-Read Path):**
    *   Feed Service queries the **Social Graph Database** to identify followed users.
    *   It queries the **Event Stream (Kafka)** for recent posts from those users.
    *   It aggregates, ranks, and merges the results.
    *   The final feed is written back to the **Distributed Cache** with a short TTL.
6.  **Feed Service $\rightarrow$ Client:** Returns the ranked feed.

## 3. Storage Choices and Data Modeling

A **Hybrid Storage Model** is essential to satisfy the conflicting consistency and scale requirements.

| Data Type | Storage Technology | Consistency Model | Scaling Strategy | Rationale |
| :--- | :--- | :--- | :--- | :--- |
| **User Profiles, Auth, Security** | **PostgreSQL (Sharded)** | Strong Consistency (ACID) | Horizontal Sharding (User ID based) | Guarantees data integrity for critical user state. |
| **Social Graph (Follows)** | **Graph Database (e.g., Neo4j)** | Strong Consistency | Partitioning/Replication | Optimized for relationship traversal (finding followers/friends). |
| **Media Files (Photos/Videos)** | **Object Storage (S3/GCS)** | Eventual Consistency | Geographic Distribution | Massive scale, cost-effective storage, high availability. |
| **Likes/Comments/Reactions** | **NoSQL (e.g., Cassandra/DynamoDB)** | Eventual Consistency | Partitioning (Post ID based) | High write throughput, high availability, and eventual consistency are acceptable. |
| **Feed Cache** | **Distributed Cache (Redis Cluster)** | Eventual Consistency | In-memory distribution | Essential for meeting the P95 < 300ms read latency target. |

**PgBouncer/Connection Pooling:** PgBouncer will be deployed in front of the PostgreSQL shards to manage and efficiently pool database connections, preventing connection exhaustion under high load.

## 4. Feed Generation Strategy

Given the scale (500M MAU) and the P95 latency target, a **Hybrid Fanout Strategy** is required.

1.  **Fanout-on-Write (For Celebrities/High-Volume Users):**
    *   When a celebrity posts, the event is published to Kafka.
    *   A dedicated **Fanout Service** consumes this event and immediately writes the post ID into the pre-computed feed caches of all 100M+ followers.
    *   **Trade-off:** Excellent read performance, but extremely high write load on the Fanout Service and potential for write bottlenecks if a celebrity posts at peak time.

2.  **Fanout-on-Read (For Average Users):**
    *   For standard users, the home feed is generated on demand.
    *   The Feed Service queries the **Social Graph DB** to find the user's followed accounts.
    *   It then queries the **Event Stream (Kafka)** for the latest posts from those accounts.
    *   The results are merged, ranked (using ML/ranking algorithms), and returned.
    *   **Trade-off:** Lower write load, but higher read latency (though mitigated by aggressive caching).

## 5. Cross-Cutting Concerns

### 5.1 Rate Limiting and Backpressure
*   **API Gateway Level (Global Rate Limiting):** Use a **Token Bucket** algorithm distributed across the gateway layer to enforce global limits per IP/API Key, protecting against DDoS and basic abuse.
*   **Service Level (Per-User/Per-Endpoint):** Implement **Sliding Window Counters** within the Feed Service and Upload Service to enforce per-user quotas (e.g., max 10 uploads/hour, max 100 likes/minute).
*   **Backpressure:** Implement **Bounded Queues** on all critical ingestion paths (Media Uploads, Feed Generation). If a queue depth exceeds a threshold, the producer (API Gateway) must be configured to **Fail Fast** (return HTTP 429) rather than allowing the queue to grow unbounded and exhaust memory.

### 5.2 Reliability and Failure Control
*   **Circuit Breakers:** Implement the **Circuit Breaker Pattern** (e.g., using a Service Mesh like Istio) between all microservices. This prevents a slow or failing service (e.g., the Recommendation Engine) from causing cascading failures in the core feed service.
*   **Retries:** Implement **Retries with Exponential Backoff and Jitter** for all external service calls. This handles transient network errors gracefully.
*   **Dead Letter Queues (DLQs):** Every critical queue (Media Processing, Feed Aggregation) must have a DLQ. Messages that fail processing repeatedly are moved to the DLQ for manual inspection and replay.
*   **Load Shedding:** Implement dynamic load shedding at the API Gateway. If overall system latency crosses a critical threshold (e.g., P99 > 500ms), the gateway can temporarily reject non-essential traffic (e.g., non-critical search queries) to preserve resources for core functions (feed reads, uploads).

### 5.3 Observability Stack
A robust observability stack is non-negotiable for debugging a system of this complexity.

*   **Distributed Tracing (Mandatory):** Implement **OpenTelemetry** across all services. Every request (upload, feed read) must carry a unique **Correlation ID** that is propagated through the API Gateway, Kafka, and all downstream services. This is the *only* way to debug the P95 < 300ms latency goal.
*   **Metrics (RED/USE):** Use **Prometheus and Grafana** to monitor the Golden Signals: **Rate** (QPS), **Errors** (Error Rates), and **Duration** (Latency) for every service and database shard.
*   **Logging:** Use **Structured Logging** (JSON format) with the Correlation ID included in every log line, allowing engineers to trace a single user journey across the entire distributed system.
*   **Alerting:** Configure alerts based on **Service Level Objectives (SLOs)**, not just static thresholds. Alert when the error budget is being consumed too quickly, or when queue depth exceeds safety limits.

## 6. Rollout and Deployment Plan

A phased, risk-mitigated rollout is essential for a system of this magnitude.

1.  **Phase 1: Infrastructure Foundation (IaC & Core Services):**
    *   Implement **Infrastructure as Code (Terraform)** for all cloud resources.
    *   Deploy the core, strongly consistent services (Authentication, User Profile) in a single region.
    *   Establish the core data pipelines (Kafka cluster, Object Storage).
2.  **Phase 2: Media Ingestion Pipeline:**
    *   Deploy the Media Upload Service and the Object Storage integration.
    *   Implement the full asynchronous processing pipeline (Upload $\rightarrow$ Kafka $\rightarrow$ Transcoding).
    *   **Rollout Strategy:** **Canary Deployment** for the upload service, routing 1% of traffic to the new version to validate the 1s acknowledgement SLA and media processing pipeline health.
3.  **Phase 3: Feed System Rollout:**
    *   Deploy the Social Graph DB and the Feed Service.
    *   **Rollout Strategy:** **Blue/Green Deployment** for the Feed Service. Route 100% of read traffic to the new Green environment only after rigorous load testing confirms P95 < 300ms under simulated peak load.
4.  **Phase 4: Global Expansion and Resilience Hardening:**
    *   Deploy the entire stack across **Multi-Region Active/Active** configuration.
    *   Introduce advanced resilience patterns (Chaos Engineering, full Circuit Breaker mesh).
    *   Gradually shift the Fanout strategy from Fanout-on-Read to a hybrid model based on real-world performance metrics.
