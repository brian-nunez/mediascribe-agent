Design for a global, real-time collaborative document editor requires a highly distributed, event-driven architecture that prioritizes availability and low latency, leveraging Conflict-free Replicated Data Types (CRDTs) for conflict resolution and a hybrid persistence strategy for offline capabilities.

### 1. Core Architectural Decision: CRDTs vs. Operational Transformation (OT)

**Recommendation:** Adopt **Conflict-free Replicated Data Types (CRDTs)** as the primary synchronization mechanism.

**Rationale & Tradeoffs:**
*   **CRDTs:** Are inherently designed for distributed, asynchronous environments. They allow concurrent edits from multiple users to be merged deterministically without requiring a central server to serialize and transform every operation. This simplifies the server logic, improves resilience, and naturally supports offline editing (as local operations can be applied and merged later).
*   **OT:** Requires a central server to maintain a canonical state and transform incoming operations against concurrent operations. This introduces significant server-side complexity, potential for complex state management, and higher latency for remote edits, which conflicts with the P95 remote edit propagation target of <300ms.

**Implementation Note:** The system should use CRDTs (e.g., RGA, LWW-Register, or specialized text CRDTs) to manage the document state on the client and propagate operations via an event stream.

### 2. End-to-End Request/Data Flow

The system follows a layered approach, separating the low-latency real-time path from the durable persistence path.

**Client $\rightarrow$ Edge $\rightarrow$ Real-Time Service $\rightarrow$ Event Stream $\rightarrow$ Persistence Layer**

1.  **Client Interaction (Real-Time Path):**
    *   **User Input:** User types/edits locally.
    *   **Local Application:** The client applies the change immediately using the local CRDT state (ensuring P95 local edit acknowledgement <100ms).
    *   **Transmission:** The client sends the CRDT operation (the delta/operation) to the **API Gateway/WebSocket Server**.
    *   **Gateway/WebSocket Server:** Performs authentication, rate limiting, and validates the operation. It immediately broadcasts the operation to all other connected users for that document via the **Real-Time Service**.
    *   **Real-Time Service:** Publishes the operation onto the **Event Streaming Platform** (e.g., Kafka topic specific to the document).

2.  **Asynchronous Persistence Path (Durability):**
    *   **Event Stream (Kafka):** The operation event is durably stored in Kafka.
    *   **Persistence Consumer Service:** A dedicated service consumes the event stream. This service is responsible for:
        *   Applying the operation to the **Primary Database** (for strong consistency of the final state).
        *   Persisting the operation log for **Version History**.
        *   Updating the **Offline Persistence Store** (for offline editing recovery).
    *   **Read Model Update:** The consumer updates read-optimized models (e.g., a denormalized document store or search index) for fast retrieval.

3.  **Remote Edit Propagation:**
    *   Receiving clients consume the operation from the Event Stream and apply it to their local CRDT state, achieving the P95 remote edit propagation target (<300ms).

### 3. Storage Choices and Data Model

| Data Type | Storage Choice | Consistency Model | Rationale |
| :--- | :--- | :--- | :--- |
| **Active Document State** | Distributed Cache (Redis Cluster) + Primary Database (PostgreSQL/CockroachDB) | Strong/Eventual Hybrid | Redis holds the current, highly-frequently accessed CRDT state for low-latency reads. PostgreSQL (or a distributed SQL like CockroachDB) serves as the source of truth for committed states and transactional metadata (permissions, user accounts). |
| **Version History/Audit Log** | Event Streaming Platform (Kafka) + Object Storage (S3/GCS) | Durable/Eventual | Kafka provides the immutable, ordered log of *every* operation. S3 stores snapshots or the raw event stream for long-term, cheap storage and replayability. |
| **Offline Edits** | Client-side Storage (IndexedDB) + Dedicated Write-Behind Queue (e.g., Redis Stream) | Eventual | Local edits are stored locally. When connectivity returns, the client sends a batch of pending operations to the Persistence Consumer Service, which applies them to the main state. |
| **Presence/Cursor** | Distributed Cache (Redis) | Eventual | Cursor positions and typing indicators are ephemeral and can tolerate slight staleness. |

### 4. Real-Time Sync Design (WebSockets & Event Streams)

*   **Protocol:** Use **WebSockets** for persistent, bidirectional, low-latency communication between the client and the WebSocket server.
*   **Flow:** The WebSocket server acts as a fanout mechanism. When an operation is received, it is immediately published to Kafka. Other connected clients are subscribed to the relevant Kafka topic (or receive a direct push from a dedicated WebSocket fanout service) to receive the operation and update their CRDT state.
*   **Scaling:** The WebSocket servers must be stateless, relying on Redis for session/presence mapping. Horizontal scaling is achieved by adding more WebSocket nodes behind a load balancer.

### 5. Conflict Resolution Strategy

*   **Primary Mechanism:** **CRDTs**. The data structure itself handles merging concurrent operations deterministically.
*   **Fallback/Safety Net:** For critical metadata (e.g., permission changes, user account updates), use **Strong Consistency** via the primary database transaction model. These operations are treated as separate, transactional events that must be fully committed before being broadcast.

### 6. Message Queue/Event Design

*   **Platform:** **Apache Kafka** is the ideal choice due to its durability, ordered log guarantees, high throughput, and ability to serve as the central nervous system for all state changes.
*   **Topics:** Use topic partitioning based on the document ID to ensure all operations for a single document are processed sequentially by the same consumer instance, maintaining order for CRDT convergence.
*   **Failure Handling:** Implement **Dead-Letter Queues (DLQs)** for any message that fails processing repeatedly (e.g., malformed CRDT operation) to prevent blocking the main stream.

### 7. Reliability, Failure Control, and Observability

| Pattern | Implementation Detail | Purpose |
| :--- | :--- | :--- |
| **Retries/Timeouts** | Implement exponential backoff and jitter for all external service calls (DB writes, external API calls). Strict timeouts on all internal service calls. | Prevent retry storms and manage transient network failures. |
| **Circuit Breakers** | Implement circuit breakers (e.g., using a Service Mesh or dedicated library) on the Persistence Consumer Service when interacting with the database or external services. | Isolate failing dependencies (e.g., a slow database replica) to prevent cascading failures. |
| **Backpressure** | Use **Kafka consumer lag** and **queue depth** metrics to dynamically adjust the rate at which the WebSocket servers push updates. If the persistence layer is falling behind, the WebSocket service should throttle incoming operations or signal clients to slow down. | Prevent the system from being overwhelmed by high write volume, ensuring the P95 latency targets are met under load. |
| **Observability** | **Distributed Tracing** (e.g., OpenTelemetry) is mandatory to trace an operation from the WebSocket ingress, through Kafka, to the database write, and back. **Golden Signals (Latency, Traffic, Errors, Saturation)** must be tracked per document and per service. | Rapidly diagnose where latency spikes occur (e.g., is it the network, the CRDT merge, or the database write?). |
| **Disaster Recovery (DR)** | **Active/Active Regional Deployment** for stateless services (API Gateway, WebSocket servers). **Asynchronous Replication** for the primary database across regions. Kafka topics should be replicated across regions. | Ensure regional failure does not result in data loss (RPO near zero) and maintain service availability (RTO low). |
| **Rollout Plan** | **Canary Deployments** for new CRDT logic or persistence consumers. Start with a small subset of users/documents. Use **Shadow Traffic** to test new synchronization logic against live traffic before full rollout. | Minimize blast radius for complex synchronization logic changes. |

### Summary of Key Tradeoffs

| Decision | Choice | Tradeoff |
| :--- | :--- | :--- |
| **Conflict Resolution** | CRDTs | Higher complexity in data structure design; eventual consistency for state convergence. |
| **Consistency** | Eventual Consistency (for real-time) + Strong Consistency (for metadata) | Achieves low latency and high availability, but requires careful application-level handling of stale reads. |
| **Synchronization** | WebSockets + Kafka | Excellent real-time performance, but introduces complexity in managing connection state and ensuring message ordering across distributed consumers. |
| **Persistence** | Write-Behind/Event Sourcing | Guarantees durability and replayability, but adds latency to the final commit and requires robust consumer logic. |
